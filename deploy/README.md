# deploy/

Self-contained deployment for kash-cards. Provisioning scripts are adapted from the
sister `runegate-infra` repo (provenance is tagged in each script header), but
kash-cards provisions **isolated** resources ‚Äî its own Azure VM, Key Vault, Cloudflare
config, and resource groups ‚Äî not co-tenant with the sister apps.

## Layout

```
deploy/
  provision-and-bootstrap.sh   # orchestrator: provision -> bootstrap [-> --with-deploy] (build the world)
  deploy.sh                    # modular redeploy: update/build/restart/start/stop/status/logs/schema
  config/                      # per-env infra config, selected by ENV (filled files GITIGNORED)
    .env.provision.dev.example #   infra params (sub, region, RG, VM size, KV, Cloudflare)
    .env.provision.prd.example #   prd infra (rg-kash-shared + rg-kash-prd, kv-kash-prd, named tunnel)
    .env.cloudflare.dev.example#   dev Cloudflare named-tunnel creds + ROUTES (host -> loopback)
    .env.cloudflare.prd.example#   prd Cloudflare creds + ROUTES (app/admin/api/... no -dev)
    sites.json                 # the 12 IIS sites: ports, host prefixes, WCF/connstr rewrites
  secrets/            # operator-authored, GITIGNORED (templates committed as *.example)
    .env.example      #   TEMPLATE -> copy to .env.<env> (.env.dev, .env.prd): non-secret config
    .vault.example    #   TEMPLATE -> copy to .vault.<env>: secrets (DB pw, provider keys) ‚Äî NEVER committed
    .smoke.env        #   (generated by vm-seed.ps1) smoke-test credentials ‚Äî GITIGNORED
  sql/
    kashnow-schema.dacpac      # BASELINE (v0): schema-only snapshot from the canonical DB; 38
                               #   tables. Applied ONCE to a brand-new empty DB, then frozen --
                               #   never re-diffed against a provisioned box (see "Schema migrations").
    migrations/                    # ordered, run-once schema migrations (applied after the baseline):
      0001-token-tables.sql              #   bearer-token tables
      0002-wallet-indexes.sql            #   prepaid-balance uniqueness indexes (wallet/address)
      0003-webhook-dedup-index.sql       #   per-event deposit-webhook dedup index
      0004-referral-commission-dedup-index.sql # per-order referral-commission dedup index (no double-pay)
      0005-otp-lockout-columns.sql       #   OTP/login/register brute-force lockout counters
      0006-login-lockout-columns.sql     #   password-login lockout columns
    seeds/                         # idempotent SQL seeds: reference rows, bootstrap admin, smoke user,
                                   #   + (dev-only) seed-dev-synthetic.sql display dataset
  scripts/
    load-env.ps1      # local dev: load .env.<env> + .vault.<env> into the current process env (-Env)
    check-no-secrets.sh # secret-shape guard (CI)
    provision/        # azure-vm-provision.sh, vm-bootstrap.ps1
    deploy/           # deploy-iis.ps1, inject-secrets.ps1, vm-iis-ops.ps1, vm-fetch-source.ps1,
                      #   vm-install-sqlpackage.ps1, vm-migrate.ps1, vm-write-config.ps1, vm-seed.ps1
    dev-seed/         # generate-dev-seed.ps1 (author-time: emits sql/seeds/seed-dev-synthetic.sql)
    perimeter/        # cloudflare-setup.sh, vm-install-cloudflared.ps1
    secrets/          # seed-kv-secrets.sh   (.env.<env>/.vault.<env> -> Key Vault; names '_' -> '-')
    verify/           # vm-verify.ps1        (NSG/loopback/callback/perimeter checks)
```

The dev seed is committed SQL (`deploy/sql/seeds/*.sql`) applied by `vm-seed.ps1` with
`sqlcmd -v`: structural reference rows (no secrets), the bootstrap admin, and the smoke
API user. In **dev** it also applies a **synthetic display dataset** (`seed-dev-synthetic.sql`,
produced by `scripts/dev-seed/generate-dev-seed.ps1`): ~25 fabricated users with wallets, a
chained balance ledger, cards and card transactions so the cardholder UI and admin lists look
realistic ‚Äî idempotent, `5eed%`-namespaced, **env-gated to dev**, and entirely fake (no prod
data), with one loginable demo cardholder whose real-inbox email (`SEED_DEMO_EMAIL`) receives the
login OTP. The seeder reproduces the app's reversible `Secure` cipher (AES-128-CBC) in PowerShell,
so password/secret ciphertext is computed at deploy time and never committed. Re-apply the seeds
any time with **`deploy.sh seed`** (data-only; no rebuild). `QryptoCard.Tests.Smoke` is the
tiered over-the-wire E2E (see its README).

To refresh or extend the synthetic dataset, re-run the generator and commit its output:
`pwsh -NoProfile -File deploy/scripts/dev-seed/generate-dev-seed.ps1` (deterministic ‚Äî same
input yields a byte-identical file).

## Environments (per-env config + secrets)

Every environment is selected by **`ENV`** (default `dev`) and has its OWN config and
secret files ‚Äî the runegate-infra convention. Nothing is shared between dev and prod:

The deploy `ENV` token for production is **`prd`** (dev = `dev`); it names the per-env files
and the `-prd` Azure resources, and matches `vm-seed`'s accepted set. The app-level gate
`QRYPTO_ENVIRONMENT` (in `secrets/.env.prd`) is a separate value and stays `prod`.

| Kind | dev | prd |
|------|-----|------|
| infra config | `config/.env.provision.dev` | `config/.env.provision.prd` |
| Cloudflare | `config/.env.cloudflare.dev` | `config/.env.cloudflare.prd` |
| non-secret app config | `secrets/.env.dev` | `secrets/.env.prd` |
| app secrets | `secrets/.vault.dev` | `secrets/.vault.prd` |

`seed-kv-secrets.sh` and `load-env.ps1 -Env <env>` read the `.env.<env>` / `.vault.<env>`
pair for the chosen environment. There is **no fallback** to an unsuffixed file, so a
missing `.vault.prd` can never silently seed dev values into the prd Key Vault.

> **Migrating an existing dev checkout:** the secrets files were previously unsuffixed.
> Rename them once so the per-env readers find them:
> `mv deploy/secrets/.env deploy/secrets/.env.dev` and
> `mv deploy/secrets/.vault deploy/secrets/.vault.dev` (both are gitignored; local only).

## Running the dev shakeout

```bash
cp deploy/config/.env.provision.dev.example deploy/config/.env.provision.dev   # fill blanks
cp deploy/secrets/.env.example  deploy/secrets/.env.dev                        # fill
cp deploy/secrets/.vault.example deploy/secrets/.vault.dev                     # fill (rotated/dev values)
ENV=dev ./deploy/provision-and-bootstrap.sh --with-deploy
# then: load deploy/secrets/.smoke.env and run `dotnet test QryptoCard.Tests.Smoke`
```

## Provisioning prod (ENV=prd)

```bash
cp deploy/config/.env.provision.prd.example  deploy/config/.env.provision.prd    # fill blanks
cp deploy/config/.env.cloudflare.prd.example deploy/config/.env.cloudflare.prd   # fill CF token + ROUTES
cp deploy/secrets/.env.example  deploy/secrets/.env.prd                          # fill prd config
cp deploy/secrets/.vault.example deploy/secrets/.vault.prd                       # fill ROTATED prd secrets
ENV=prd ./deploy/provision-and-bootstrap.sh --with-deploy
# subsequent app redeploys: ENV=prd ./deploy/deploy.sh update
```

Prod launches on `app/admin/api/callback/docs.s16.xyz` (temporary). The **domain cutover**
to `kash.cards` is a zone-knob flip: set `CLOUDFLARE_ZONE=kash.cards` in
`.env.provision.prd` and `PUBLIC_BASE_URL=https://app.kash.cards` in `.env.prd`,
re-seed the prd KV, re-run `cloudflare-setup.sh`, then `ENV=prd ./deploy/deploy.sh update`.
(Prerequisite: the `kash.cards` zone is added to Cloudflare and the prod API token covers it.)

## Redeploying app code (modular deploy)

`provision-and-bootstrap.sh` builds the world (~30-45 min). Once the box exists,
use **`deploy/deploy.sh`** for the common case ‚Äî redeploy app code, all tiers or
just one, without re-provisioning, without the perimeter, and (by default)
without touching the schema. It runs from your **operator devbox** (the machine
with `az` auth and the filled `config/` + `secrets/`); the VM is NSG-dark, so each
step is dispatched via `az vm run-command invoke`, same as the orchestrator.

```bash
ENV=dev ./deploy/deploy.sh update              # all tiers: fetch -> build -> inject -> recycle -> start
ENV=dev ./deploy/deploy.sh update int          # just the INT money tier
ENV=dev ./deploy/deploy.sh update --with-schema # app deploy + apply pending schema migrations
ENV=dev ./deploy/deploy.sh restart api-callback # recycle one pool (start it if stopped)
ENV=dev ./deploy/deploy.sh status              # pool/site state + port for every tier
ENV=dev ./deploy/deploy.sh logs dashboard      # tail the latest IIS log
ENV=dev ./deploy/deploy.sh start               # start-guarantee: every pool ends Started
ENV=dev ./deploy/deploy.sh sync dashboard      # front-end fast lane: push CHANGED static/markup from your LOCAL tree
ENV=dev ./deploy/deploy.sh seed                 # re-apply the seed SQL (dev also loads the synthetic dataset); data-only, no rebuild
```

Commands: `update [svc] [--with-schema]`, `build [svc]`, `sync <svc> [files...] [--recycle]`,
`restart [svc]`, `start [svc]`, `stop [svc]`, `status`, `logs [svc]`, `schema`, `seed`. A
**service alias** is the app pool minus the `kash-` prefix: `api api-public
api-admin api-callback api-scheduler apidocs dashboard dashboard-admin int
int-callback int-scheduler scrapper`.

### Front-end fast lane (`sync`)

`update` is correct but heavy for a CSS/markup tweak: it pushes through GitHub,
restores NuGet, and does a clean Release rebuild ‚Äî minutes for a change that needs
no compile. **`sync`** is the fast lane for front-end iteration on the Web Forms
tiers (`dashboard`, `dashboard-admin`):

```bash
ENV=dev ./deploy/deploy.sh sync dashboard                 # auto-detect changed files under the tier
ENV=dev ./deploy/deploy.sh sync dashboard Content/app.css # or name them explicitly
ENV=dev ./deploy/deploy.sh sync dashboard --recycle       # also recycle the pool (rarely needed)
```

It copies **changed static assets + ASP.NET markup** (`.aspx/.ascx/.master/.css/
.js`/images/‚Ä¶) straight from your **local working tree** into the live site root
(`C:\inetpub\kash-cards\<project>`) ‚Äî no GitHub push, no fetch, no restore, no
MSBuild, no secret inject. IIS serves static content immediately and recompiles
changed markup on the next request, so a tweak lands in seconds.

- **Auto-detect** = files under the tier's project that differ from
  `origin/<branch>` plus untracked ones (so committed-but-unpushed edits ship
  too). The exact file list is printed before anything is sent ‚Äî eyeball it.
- **Refuses anything that needs a compile.** If a `.cs` / `.csproj` / `.resx` /
  `packages.config` (or any `*.config`, which on the box is patched) is in the
  change set, `sync` stops and tells you to run `build`/`update` ‚Äî the fast lane
  can never silently ship a stale binary or clobber a patched `Web.config`.
- **Deletions aren't propagated** (sync only copies); it warns and points you to
  `build`/`update` if a local delete must take effect.
- Bounded to a small payload (run-command limit); a large change set fails with a
  message to use `build` instead. Override the cap with `SYNC_MAX_B64` if needed.

- **`update`** is app-only by default ‚Äî it never touches the DB. Pass
  `--with-schema` (or run `deploy.sh schema`) to apply pending schema migrations
  when a change needs one; the runner only applies what is not yet recorded, so it
  is safe to re-run. See **Schema migrations** below.
- **`update` / `build` / `restart` / `start`** all end by guaranteeing every
  targeted pool reaches the **Started** state (and fail loudly if one cannot) ‚Äî so
  a deploy can never silently leave a WCF money tier down.

The on-box halves live in `scripts/deploy/`: `deploy-iis.ps1` and
`inject-secrets.ps1` take an optional `-Service <alias>`; `vm-iis-ops.ps1` owns
the lifecycle verbs (start/stop/restart/status/logs + the start-guarantee);
`vm-sync-content.ps1` is the on-box half of `sync` (validate the payload, extract
into the site root, optional recycle).

`update`'s source step (`vm-fetch-source.ps1`) keeps a **persistent git clone** at
`C:\src\kash-cards` and fetches **incrementally**: first run clones, every run
after is `git fetch` + `git reset --hard origin/<branch>` + `git clean -ffd` ‚Äî
only the changed objects move, in seconds. `-ffd` (not `-ffdx`) deliberately keeps
gitignored content, so the `packages/` NuGet cache and the on-box `config/` survive
a fetch (no needless re-restore); `obj/`/`bin/` are cleaned per-project by
`deploy-iis.ps1`. The repo token comes from Key Vault (`REPO-TOKEN`) and is handed
to git via an environment-injected `http.extraheader` ‚Äî never written to disk, the
remote URL, or argv. (The **first** `update` after this change re-clones from
scratch, since the box still holds the old non-git source tree; subsequent ones are
incremental.)

**Prerequisites for the build-on-box fetch (all provisioned by `vm-bootstrap.ps1` /
seeding ‚Äî called out because they had never been exercised end-to-end before #65):**
- **git** is installed on the VM by `vm-bootstrap.ps1` (MinGit, portable, to
  `C:\Tools\git` ‚Äî where `vm-fetch-source.ps1`'s `Resolve-Git` looks). No system git
  needed.
- **`REPO-TOKEN`** (a GitHub read token for the private repo) must exist in Key Vault.
  It is an **operator secret**: put `REPO_TOKEN=‚Ä¶` in `deploy/secrets/.vault.<env>` and run
  `ENV=<env> deploy/scripts/secrets/seed-kv-secrets.sh` to upload it (seeded as `REPO-TOKEN`).
  `deploy.sh update` does **not** seed Key Vault ‚Äî editing `.vault.<env>` alone is not enough.
- On-box scripts resolve **`az` by full path** (`C:\Program Files\Microsoft SDKs\Azure\
  CLI2\wbin\az.cmd`), not via PATH: under `az vm run-command` the agent's PATH is
  captured at service start and does not pick up a later install.

Note: `az vm run-command invoke` returns exit 0 even when the inner script fails ‚Äî
`deploy.sh` greps the captured output for the `[xx]` fatal marker, so always read the
deploy log rather than trusting the wrapper's exit code.

## Schema migrations

The database schema is managed as a **frozen baseline + ordered, run-once migrations**, applied
by `scripts/deploy/vm-migrate.ps1` (the `schema` command, and `update --with-schema`).

- **Baseline (`sql/kashnow-schema.dacpac`)** is a desired-state snapshot of the 38 canonical
  tables. It is published **once**, to a brand-new empty database, to lay down the baseline ‚Äî
  then never touched again. It is **not** re-published against a provisioned box: a declarative
  publish diffs the live DB against the snapshot and renovates it to match, which would (a) DROP
  the lockout columns the migrations add and (b) widen `TXID`/`Address` back to `nvarchar(max)`,
  breaking the money-safety unique indexes the migrations create. Both are deliberate divergences
  from the snapshot, so the snapshot must not be allowed to undo them.
- **Migrations (`sql/migrations/NNNN-*.sql`)** are hand-written, additive, internally
  `IF NOT EXISTS`-guarded DDL, applied in filename order. Each one that runs is recorded in a
  ledger table, **`dbo.SchemaMigrations`**, so it never runs twice.
- **`vm-migrate.ps1`** decides per-box: a database that does not exist (or has zero user tables)
  is **fresh** ‚Üí publish the baseline, then run all migrations; a database that already has tables
  is **adopted** ‚Üí skip the baseline entirely and run only migrations not yet in the ledger.
  Because the migrations are `IF NOT EXISTS`-guarded, the first run on an already-provisioned box
  simply replays them as no-ops and populates the ledger ‚Äî no manual baselining.

**Adding a schema change:** drop a new `sql/migrations/NNNN-<name>.sql` (next number, additive and
`IF NOT EXISTS`-guarded), commit it, and run `ENV=dev ./deploy/deploy.sh schema`. Never edit the
baseline dacpac or an already-applied migration; always add a new one.

## Scheduled jobs

The **reconciliation sweep** recovers card spends stranded at `pending provider` (the balance
was debited but the WasabiCard outcome was ambiguous). It runs on a recurring **Windows
scheduled task** registered by `vm-bootstrap.ps1` (Step 6):

- Task `KashCards-ReconcilePending`, SYSTEM, **every 10 min**, `-MultipleInstances IgnoreNew`.
- It invokes `deploy/scripts/scheduler-trigger.ps1 -VaultName <kv>`, which reads
  `SCHEDULER-SHARED-SECRET` from Key Vault (via the VM managed identity) and **POSTs** to the
  API.Callback loopback endpoint `http://127.0.0.1:8084/v1/payment/reconcile/pending` with an
  `X-Scheduler-Auth` header. Success/failure is written to the `KashCardsScheduler` Application
  event-log source.

**Security note:** the API.Callback host is publicly reachable via the Cloudflare tunnel, so the
endpoint is gated by the shared secret (constant-time, fail-closed) ‚Äî not network isolation.
Keep `SCHEDULER_SHARED_SECRET` strong and rotate it with the rest of the Key Vault material.

Verify after a deploy:
```powershell
Get-ScheduledTask -TaskName 'KashCards-ReconcilePending' | Format-Table TaskName, State
Start-ScheduledTask -TaskName 'KashCards-ReconcilePending'
Get-EventLog -LogName Application -Source KashCardsScheduler -Newest 5
```

## How secrets reach the app

The app reads secrets from **process environment variables only** (see
`QryptoCard.Sec/SecretsConfig.cs`). It never reads files or Key Vault directly.

- **Local / dev:** `deploy/scripts/load-env.ps1 -Env <env>` loads `secrets/.env.<env>` +
  `secrets/.vault.<env>` into the current process before you run the app (default `dev`).
- **Server:** `inject-secrets.ps1` pulls from Azure Key Vault (via the VM managed
  identity) and writes per-app-pool env vars into `applicationHost.config`.

So the same code works everywhere; only the source of the env vars differs.
